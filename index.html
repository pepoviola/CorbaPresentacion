Common Object Request Broker Architecture

Hoy vamos a hablar de

Object Request Broker

Introducción

Características
  • Es un sistema autodescriptivo
  • ORB ofrece la conectividad en un sistema CORBA
  • Canal de comunicación a través del cual los objetos solicitan servicios con independencia de su ubicación fésica
  • Permite la invocación estática y dinámica
  • Permite una comunicación transparente cuando un cliente envía una solicitud a un objeto remoto

    • Arquitectura

    Visión General

    Arquitectura
    Detalle de la Arquitectura (cliente)
  • IDL Stubs: Funciones generadas desde la interfaz IDL para “enlazarlas” a los clientes Provee una interfaz de invocación estática
  • Dynamic Invocation Interface (DII): Permite especificar y construir requerimientos en tiempo de ejecución. Operaciones: create_request, invoke, send, get_response
    • Detalle de la Arquitectura (Servidor)
  • IDL Skeleton: Funciones generadas desde la interfaz IDL para “enlazarlas” a las implementaciones de objetos
  • Dynamic Skeleton Interface (DSI): Análogo al DII del lado de la implementación de objetos. Puede recibir invocaciones estáticas o dinámicas desde los clientes
    • Detalle de la Arquitectura
  • ORB Interface:Provee funciones para acceder directamente al ORB core desde los clientes y desde las implementaciones de objetos

    Su interfaz no depende de la interfaz de los clientes ni de las interfaces de las implementaciones de objetos

  • Object Adapter: Provee funciones para instanciar objetos, pasar requerimientos y manipular referencias de objetos

    Provee inter-operabilidad

    • Detalle de la Arquitectura
  • Repositorio de Interfaces (Interface Repository): Provee persistencia de objetos

    Su información permite que un programa encuentre un objeto cuya interfaz no conoce en tiempo de compilación

  • Repositorio de Implementaciones (Implementation Repository): Contiene información que permite al ORB core localizar y activar implementaciones de objetos

    • ORB (servidor)

    Skeleton, Dynamic Skeleton Interface, Object Adapter

    Skeleton
  • Representa el elemento análogo al stub del cliente.
  • Se encarga de simular la petición remota como una petición local a la implementación real del objeto.
  • Para el servidor todas las llamadas parecen locales.
  • Generado a partir del IDL por un compilador IDL.
  • Realiza el unmarshalling de las invocaciones del cliente.
  • Proporcionan interfaces para el acceso estático a cada uno de los servicios proporcionados por el servidor.
    • DSI (Dynamic Skeleton Interface)
  • Es el equivalente al DII del lado del servidor.
  • Permite al servidor recibir la invocación en cualquier objeto, incluso en aquellos en donde no se conoce la interfaz IDL (en tiempo de ejecución).
  • El servidor no tiene que estar vinculado con el código skeleton.
    • Object Adapters
  • Son las claves de la portabilidad y la flexibilidad en la implementación de los servidores.
  • Permite preparar los objetos a la recepción de las peticiones e informar al ORB cuando estas peticiones han sido ejecutadas.
    • Object Adapters (continuación)
  • Las funciones que debe realizar el adaptador de objetos son:
  • Generación e interpretación de las referencias a objetos.
  • Invocación de métodos apropiados cuando llega una petición remota (upcall).
  • Seguridad en las interacciones.
  • Activación y desactivación de objetos e implementaciones.
  • Traducción de referencias a objetos a sus correspondientes implementaciones.
  • Registro objetos en el repositorio de implementaciones.

    • Marshalling/ Unmarshalling

    Marshaling / Unmarshaling
  • Acción de Marshalling: traducción de parámetros de entrada.
  • Acción de Unmarshalling: conversión de parámetros de la red.
    • Marshaling
  • Conversión de formato en plataforma independiente.
    • Unmarshaling
  • Conversión de formato de red en plataforma específica.
